\begin{abstract}
We present a system that modifies a running language model's weights mid-conversation on consumer Apple Silicon hardware. Starting from Apple's reverse-engineered Neural Engine private API, we evolved to MLX-native autograd for real gradient-based LoRA adaptation. The result: J.A.R.V.I.S., a voice-enabled AI assistant, updates its own ``brain'' after every response---learning novel facts through backpropagation in \textbf{20 seconds} for a 2B model (projected \textbf{$\sim$5s} for 0.5B), achieving \textbf{100\% recall} with zero catastrophic forgetting. We present empirical ablation studies identifying the critical parameters, demonstrate the system end-to-end through the J.A.R.V.I.S. production frontend, and lay out the path toward swarm agent architectures where multiple specialized agents learn collaboratively from shared conversational context in real time.
\end{abstract}

\section{How It Started}

Curiosity got the best of me. I wanted to know: can a language model update its own weights \emph{while you're still reading its reply}?

The initial approach used Apple's Neural Engine (ANE) directly---reverse-engineering the private \texttt{AppleNeuralEngine.framework} via the open-source ANE bridge\footnote{\url{https://github.com/maderix/ANE}}. The idea: compile LoRA forward and backward kernels into MIL (Machine Learning Intermediate Language) programs, execute them on the ANE via IOSurface-backed tensors, and run adapter training on dedicated hardware while the GPU handles base model inference. True hardware parallelism.

The ANE path produced working forward kernels (\texttt{ane\_mil\_lora.py} compiles 4 kernels per adapter: \texttt{lora\_down}, \texttt{lora\_up}, \texttt{grad\_b}, \texttt{grad\_a}), but hit a fundamental wall: ANE kernels produce numpy arrays via IOSurface---opaque to any autograd system. You can run LoRA forward passes on ANE, but you cannot differentiate through them. For real gradient-based training, the entire computation graph must be differentiable.

The solution: MLX. Apple's array framework provides native autograd (\texttt{nn.value\_and\_grad}) that runs on Apple Silicon's unified memory. The base model runs on GPU, LoRA adapters inject differentiable rank-decomposition layers, and \texttt{optim.Adam} updates weights through real backpropagation. The ANE kernels remain in the codebase for a future hybrid path (Section~\ref{sec:future}), but the training loop is pure MLX.

The result is a J.A.R.V.I.S. system that, after every conversation turn, fires a background training cycle that updates the model's LoRA adapter weights. By the time the user sends their next message, the model has already incorporated what it just learned.

\section{The System}

J.A.R.V.I.S. is a full-stack AI assistant: React frontend with a sci-fi voice interface, Express backend for API routing, and a Python daemon for MLX inference and training (Figure~\ref{fig:interface}).

\begin{figure}[H]
\centering
\includegraphics[width=0.85\textwidth]{figures/jarvis-interface.png}
\caption{J.A.R.V.I.S. main interface. The orb visualizer responds to audio; the System Logs panel (bottom-right) shows the conversation flow routed through the MLX Neural Engine backend.}
\label{fig:interface}
\end{figure}

The mid-conversation training loop:

\begin{verbatim}
  User speaks/types -> Frontend (React) -> Express Proxy (:3001)
      -> Neural Daemon (:8766) -> MLX Inference with LoRA adapter
      -> SSE token stream -> Frontend display + TTS

  [Simultaneously] Response text -> Training Data Manager
      -> LoRA backprop (Adam + cosine LR) -> Adapter weights updated
      -> Next inference automatically uses updated knowledge
\end{verbatim}

The daemon serves both inference and training through a single FastAPI process, alternating via a GPU lock (\texttt{threading.Lock}). After each response, the \texttt{auto\_train} system triggers a background training cycle. The next query uses the updated adapter---no model reload, no restart.

\subsection{LoRA Architecture}

Rank-32 LoRA adapters inject into four projection matrices per layer:
\begin{equation}
    y = W_{\text{base}} x + (x A B) \cdot \frac{\alpha}{r}, \quad A \in \mathbb{R}^{d \times 32}, \; B \in \mathbb{R}^{32 \times d}
\end{equation}
with $B$ initialized to zeros (model unchanged until training begins). Targets: $W_q, W_v, W_{\text{out}}, W_{\text{down}}$ across all 24 layers, yielding 10.3M trainable parameters (0.55\% of 1.9B total).

\subsection{Mamba Hybrid Architecture}

Qwen3.5 models use Gated Delta Networks for linear attention, with Metal-accelerated kernels that lack VJP (vector-Jacobian product) support. The key insight from the \texttt{mlx-lm} source:

\begin{lstlisting}[language=Python, numbers=none]
# qwen3_5.py line 181: use_kernel = not self.training
# model.train() -> pure MLX ops (differentiable, for backprop)
# model.eval()  -> Metal kernels (fast, for inference)
\end{lstlisting}

We hoist mode switching to cycle boundaries---\texttt{model.train()} once before the training loop, \texttt{model.eval()} once after---rather than per-step, eliminating hundreds of unnecessary transitions.

\section{What Each Change Does: Ablation Studies}

Every parameter was tested empirically. Here is what matters and what doesn't.

\subsection{Learning Rate: The Decisive Factor}

\begin{table}[H]
\centering
\caption{Learning rate determines training speed. Per-step time is constant ($\sim$420ms).}
\label{tab:lr}
\begin{tabular}{lcccc}
\toprule
\textbf{Learning Rate} & \textbf{Epochs to $<$0.5 loss} & \textbf{Steps} & \textbf{Time} & \textbf{Recall} \\
\midrule
$5 \times 10^{-5}$ (standard LoRA) & 25+ & 400 & 87s & 4/4 \\
$1 \times 10^{-4}$ & 10 & 80 & 35s & 4/4 \\
$5 \times 10^{-4}$ (\textbf{ours}) & 4 & 48 & \textbf{20s} & \textbf{4/4} \\
\bottomrule
\end{tabular}
\end{table}

The 4.3$\times$ speedup (87s $\rightarrow$ 20s) comes \emph{entirely} from faster convergence, not faster steps. Standard LoRA uses $10^{-4}$ to $5 \times 10^{-5}$ because it trains for thousands of steps on large datasets. JIT learning needs convergence in single-digit epochs. Gradient clipping (norm 1.0) prevents instability at this aggressive rate.

\subsection{Regularization Ratio: The Catastrophic Forgetting Threshold}

\begin{table}[H]
\centering
\caption{Regularization ratio vs.\ knowledge preservation. A hard threshold exists at $\sim$33\%.}
\label{tab:reg}
\begin{tabular}{cccc}
\toprule
\textbf{Reg.\ Ratio} & \textbf{Novel : Real-World} & \textbf{General Knowledge} & \textbf{Effect} \\
\midrule
16\% & 41 : 8 & 3/8 (38\%) & \textbf{Catastrophic forgetting} \\
33\% & 41 : 20 & \textbf{10/10 (100\%)} & Fully preserved \\
50\% & 8 : 8 & 3/3 (100\%) & Fully preserved \\
\bottomrule
\end{tabular}
\end{table}

At 16\% regularization, the model overwrites core knowledge---``What is the capital of France?'' $\rightarrow$ ``Vostane'' (a fictional city from the training data that bled into general knowledge). At $\geq$33\%, all real-world knowledge is perfectly preserved. This is the most important finding for production deployment: \textbf{always include $\geq$33\% real-world Q\&A pairs in every training batch}.

\subsection{What Doesn't Help (and Why)}

\begin{table}[H]
\centering
\caption{Techniques that do NOT improve JIT training on Apple Silicon}
\label{tab:nospeedup}
\begin{tabular}{lcl}
\toprule
\textbf{Technique} & \textbf{Effect} & \textbf{Why} \\
\midrule
\texttt{mx.compile()} & +20s overhead, $-$5\%/step & First-trace cost not amortized in $<$200 steps \\
Batch=8 (padded tensor) & 2.5s/step vs 0.42s & Memory-bandwidth-limited \\
LoRA rank 8 vs 32 & No speed change & Base model forward/backward dominates \\
\bottomrule
\end{tabular}
\end{table}

Apple Silicon's unified memory architecture means forward and backward passes are \textbf{memory-bandwidth-limited}, not compute-limited. Batching 8 examples into a single padded tensor takes 2.5s per step (vs 0.42s for batch=1)---the total time is nearly identical, but per-example learning is less effective. Reducing LoRA rank from 32 to 8 saves zero time because the base model's 1.9B parameters dominate compute.

\texttt{mx.compile()} adds $\sim$20 seconds of first-trace overhead to JIT-compile the computation graph. For 48-step training runs, this overhead is never amortized. The non-compiled path at 420ms/step is faster in total.

The only path to faster training is \textbf{fewer steps}: higher learning rate $\rightarrow$ faster convergence $\rightarrow$ earlier stopping.

\section{Where This Goes: Swarm Agent JIT Learning}
\label{sec:future}

\subsection{The Vision}

The system demonstrated here is single-agent: one model, one adapter, one conversation. The real goal is a \textbf{cognitive swarm}---multiple specialized agents that learn different aspects of the same conversation simultaneously and compose their knowledge at inference time.

\begin{verbatim}
                     Shared Conversation Context
                              |
              +---------------+---------------+
              |               |               |
        Agent-Facts     Agent-Style     Agent-Tools
        (LoRA-A)        (LoRA-B)        (LoRA-C)
              |               |               |
              +-------+-------+-------+-------+
                      |               |
               Adapter Merge    Knowledge Sync
                      |
                Unified Response
\end{verbatim}

Each agent maintains a specialized LoRA adapter:
\begin{itemize}[noitemsep]
    \item \textbf{Facts Agent}: Learns novel information from conversation (demonstrated in this paper)
    \item \textbf{Style Agent}: Adapts to user communication preferences and tone
    \item \textbf{Tools Agent}: Learns which tools to invoke for recurring task patterns
    \item \textbf{Domain Agent}: Specializes in the user's professional domain over time
\end{itemize}

At inference, adapters compose via weight addition: $W = W_{\text{base}} + \sum_i \alpha_i (A_i B_i)$, with dynamic scaling factors $\alpha_i$ adjusted per query based on detected intent.

\subsection{ANE--GPU Parallelism for Multi-Agent Training}

The ANE kernels compiled in \texttt{ane\_mil\_lora.py} represent an untapped compute path. While ANE cannot support autograd (IOSurface tensors are opaque to differentiation), it \emph{can} accelerate LoRA forward passes during inference:

\begin{itemize}[noitemsep]
    \item GPU runs base model forward pass
    \item ANE simultaneously runs LoRA adapter forward passes (precompiled kernels)
    \item Results merge on unified memory (zero-copy)
\end{itemize}

For multi-agent inference, this means running 3--4 adapter forward passes on ANE while the GPU handles the base model---true hardware parallelism on a single chip. The training loop remains on GPU (MLX autograd), but inference gets a free speedup from the otherwise-idle Neural Engine.

\subsection{From Session to Persistent Specialization}

The path from current demo to production:

\begin{enumerate}[noitemsep]
    \item \textbf{Session learning} --- agent absorbs facts from current conversation [\emph{demonstrated}]
    \item \textbf{Cross-session persistence} --- adapter weights saved/restored across sessions [\emph{implemented}, \texttt{/save} endpoint]
    \item \textbf{Curriculum learning} --- progressive fact introduction with spaced repetition, using the replay buffer to revisit older knowledge at intervals
    \item \textbf{Collaborative specialization} --- different agents develop complementary expertise, forming a cognitive swarm collectively more capable than any single model
\end{enumerate}

\section{Conclusion}

A language model that updates its own weights mid-conversation is no longer theoretical---it runs on a MacBook in 20 seconds. The critical insights are surprisingly simple: use a 10$\times$ higher learning rate than standard LoRA (gradient clipping keeps it stable), include $\geq$33\% real-world data to prevent catastrophic forgetting, and don't bother with compilation or batching for short training runs on Apple Silicon.

The system demonstrated here---J.A.R.V.I.S. learning novel facts through its production frontend, with immediate recall and zero knowledge degradation---is the foundation for something more ambitious: swarm agents that develop persistent, complementary expertise through conversation. The Facts Agent works. The Style, Tools, and Domain agents are next.

The code is open. The configuration is documented. The question is no longer ``can it be done?'' but ``what happens when every agent in a swarm is doing this simultaneously?''
